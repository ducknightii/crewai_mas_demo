# 产品设计 SOP（标准操作规程）草案

## 适用场景
本SOP适用于新产品功能开发或现有产品重大迭代的设计阶段，确保产品规格文档的完整性和可执行性。

## 角色分工
- **PM（产品经理）**：负责撰写和维护产品规格文档，确保需求准确转化为可执行的功能规格。
- **Manager**：负责提供初始需求文档，并对最终产品规格进行验收。

## 执行步骤

### 第一步：需求理解
1. 读取需求文档 `/mnt/shared/needs/requirements.md`
2. 确认以下内容已完整：
   - 目标与核心诉求
   - 边界（本次做什么 / 不做什么）
   - 约束条件
   - 验收标准

### 第二步：产品规格文档撰写
产品规格文档路径：`/mnt/shared/design/product_spec.md`

必须包含以下章节：

#### 1. 产品概述
- 背景与目的
- 目标用户
- 核心价值

#### 2. 用户故事
格式：作为 [用户角色]，我希望 [功能描述]，以便 [目标/价值]

#### 3. 功能规格
每个功能按以下格式描述：
```
## F-01：[功能名称]
- 描述：[详细说明]
- 输入：[用户输入/触发条件]
- 输出：[系统响应/结果]
- 验收标准：[ ] 具体可测试的标准
```

#### 4. 验收标准（DoD）
所有功能的验收标准汇总，格式为可勾选的清单。

### 第三步：通知 Manager
文档完成后，通过 mailbox-ops skill 向 Manager 发送完成通知：
- `to: manager`
- `from_: pm`
- `type_: task_done`
- `subject: 产品文档已完成`
- `content: 产品规格文档路径（/mnt/shared/design/product_spec.md）及简要说明`

## Checkpoint
- [ ] 需求文档已完整读取并理解
- [ ] 产品概述清晰明确
- [ ] 用户故事覆盖所有关键场景
- [ ] 功能规格描述无歧义
- [ ] 验收标准可量化、可测试

## 质量标准
- 功能规格必须可执行、可测试
- 验收标准必须量化，不得使用模糊表述
- 文档长度：正式产品不低于 500 字
- 所有功能点必须有对应的验收标准
